Page Transitions

Page transitions are a powerful way to enhance the user experience in your Dioxus application. They provide visual continuity between route changes, making your app feel more polished and professional. Dioxus Motion makes implementing these transitions simple and declarative.

Why Use Page Transitions?

  • Improved User Experience: Smooth transitions between pages reduce the jarring effect of instant content changes.
  • Visual Hierarchy: Different transition types can convey navigation direction and relationship between pages.
  • Professional Polish: Well-implemented transitions add a level of refinement that users associate with high-quality applications.

How It Works

Dioxus Motion's page transitions work by intercepting route changes and applying animations during the transition. The library handles all the complexity of managing the animation timing and ensuring both the entering and exiting pages are properly animated.

1

Define Transitions

Add transition attributes to your route definitions to specify which animation to use for each route.

2

Use AnimatedOutlet

Replace the standard Outlet component with AnimatedOutlet in your layout to enable the transitions.

3

Navigate Normally

Use standard Dioxus navigation. The transitions will automatically apply when routes change.

Implementation Steps

1. Enable Transitions Feature

Add the transitions feature to your dioxus-motion dependency in Cargo.toml:

dioxus-motion = { git = "https://github.com/wheregmis/dioxus-motion.git", branch = "main", default-features = false, optional = true }

[features]
default = ["web"]
web = ["dioxus/web", "dioxus-motion/web", "dioxus-motion/transitions"]
desktop = [
    "dioxus/desktop",
    "dioxus-motion/desktop",
    "dioxus-motion/transitions",
]
mobile = ["dioxus/mobile", "dioxus-motion/desktop", "dioxus-motion/transitions"]

2. Add MotionTransitions Derive

Add the MotionTransitions derive macro to your Route enum:

#[derive(Routable, Clone, Debug, PartialEq, MotionTransitions)]
#[rustfmt::skip]
enum Route {
    #[layout(NavBar)]
        #[route("/")]
        #[transition(Fade)]
        Home {},
        #[route("/slide-left")]
        #[transition(ZoomIn)]
        SlideLeft {},
        #[route("/slide-right")]
        SlideRight {},
        #[route("/slide-up")]
        SlideUp {},
        #[route("/slide-down")]
        SlideDown {},
        #[route("/fade")]
        Fade {},
    #[end_layout]
    #[route("/:..route")]
    PageNotFound { route: Vec },
}

3. Use AnimatedOutlet

Replace Outlet with AnimatedOutlet in your layout component:

#[component]
fn NavBar() -> Element {
    rsx! {
        nav { id: "navbar",
            Link { to: Route::Home {}, "Home" }
            Link { to: Route::SlideLeft {}, "Blog" }
        }
        AnimatedOutlet:: {}
    }
}

Configurable Animation Context

Dioxus Motion allows you to customize the spring physics used in transitions through Dioxus's context system. This provides fine-grained control over animation behavior without modifying individual transition definitions.

Basic Context Setup

Provide custom spring configuration to all page transitions using a context provider:

use dioxus::prelude::*;
use dioxus_motion::prelude::*;

fn main() {
    dioxus::launch(|| {
        rsx! {
            head {
                link { rel: "stylesheet", href: MAIN_CSS }
            }
            App {}
        }
    });
}

#[component]
fn App() -> Element {
    // Define custom spring configuration
    let spring = use_signal(|| Spring {
        stiffness: 200.0,  // Higher = faster, snappier
        damping: 30.0,     // Higher = less bounce
        mass: 0.8,         // Lower = more responsive
        velocity: 0.0,     // Initial velocity
    });

    // Provide spring context to all child components
    use_context_provider(|| spring);

    rsx! {
        Router:: {}
    }
}

Spring Parameters

Stiffness (default: 160.0)

Controls how quickly the animation reaches its target. Higher values = faster, more aggressive motion.

Damping (default: 25.0)

Controls oscillation and bounce. Higher values = smoother, less bouncy motion.

Mass (default: 1.0)

Controls inertia. Higher values = more sluggish, weighty motion.

Velocity (default: 0.0)

Initial velocity. Can create pre-existing motion effects.

Common Configurations

Bouncy (Fun & Playful)

stiffness: 180.0, damping: 15.0

Smooth (Professional)

stiffness: 140.0, damping: 35.0

Fast (Snappy)

stiffness: 220.0, damping: 20.0, mass: 0.8

Gentle (Subtle)

stiffness: 120.0, damping: 30.0, mass: 1.3

Multiple Animation Contexts

You can provide different spring configurations for different parts of your application:

#[component]
fn AdminSection() -> Element {
    // Faster, more aggressive animations for admin interfaces
    let admin_spring = use_signal(|| Spring {
        stiffness: 220.0,
        damping: 20.0,
        mass: 0.8,
        velocity: 0.0,
    });

    use_context_provider(|| admin_spring);

    rsx! {
        AnimatedOutlet:: {}
    }
}

#[component]
fn UserSection() -> Element {
    // Gentler animations for user-facing content
    let user_spring = use_signal(|| Spring {
        stiffness: 140.0,
        damping: 35.0,
        mass: 1.0,
        velocity: 0.0,
    });

    use_context_provider(|| user_spring);

    rsx! {
        AnimatedOutlet:: {}
    }
}

Pro Tip: Context Fallback

If no spring context is provided, Dioxus Motion automatically falls back to sensible defaults. This means transitions work out-of-the-box while still allowing customization when needed.

Best Practices

Follow these guidelines to create effective and performant page transitions in your Dioxus application.

Be Consistent

Use consistent transition patterns throughout your application. For example, use SlideLeft for forward navigation and SlideRight for backward navigation to create a cohesive experience.

Keep Transitions Short

Aim for transitions between 200-300ms. Longer transitions can make your app feel sluggish, while shorter ones might be too abrupt. The default durations are optimized for most cases.

Consider Mobile Performance

On mobile devices, prefer simpler transitions like Fade or SlideLeft/Right. Complex animations with multiple properties might cause performance issues on lower-end devices.

Use Semantic Transitions

Choose transitions that match the mental model of your navigation. For hierarchical navigation, consider using ZoomIn for drilling down and ZoomOut for going back up a level.

Example with Nested Routes

#[derive(Routable, Clone, Debug, PartialEq, MotionTransitions)]
#[rustfmt::skip]
enum Route {
    #[layout(NavBar)]
        #[route("/")]
        #[transition(SlideDown)]
        Home {},

        #[nest("/blog")]
        #[layout(Blog)]
            #[route("/")]
            #[transition(SlideUp)]
            BlogList {},

            #[route("/:name")]
            #[transition(SlideRight)]
            BlogPost { name: String },

        #[end_layout]
        #[end_nest]

    #[end_layout]

    #[route("/:..route")]
    #[transition(Fade)]
    PageNotFound { route: Vec },
}

Troubleshooting

If you encounter issues with your page transitions, here are some common problems and their solutions.

Transitions Not Working

If your transitions aren't working at all, check the following:

  • Verify that you've enabled the transitions feature in your Cargo.toml
  • Ensure you've added the MotionTransitions derive to your Route enum
  • Confirm you're using AnimatedOutlet instead of the standard Outlet
  • Check that you've specified transition attributes for your routes

Flickering or Jumpy Transitions

If your transitions appear to flicker or jump:

  • Ensure your page components have consistent dimensions to avoid layout shifts
  • Try using simpler transitions like Fade instead of complex ones
  • Check for any CSS that might be interfering with the animations

Performance Issues

If transitions are causing performance problems:

  • Reduce the complexity of your page components
  • Use simpler transitions for mobile devices
  • Ensure you're not animating too many properties simultaneously
  • Consider using hardware acceleration with CSS transforms where possible
